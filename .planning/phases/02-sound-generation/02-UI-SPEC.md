## Color

> Phase 2 inherits the color contract from Phase 1 UI-SPEC. One new semantic color is introduced for the active/playing state.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #111111 | Page background (same as Phase 1 overlay background) |
| Secondary (30%) | #1e1e1e | App body background `#app` (same as Phase 1) |
| Accent (10%) | #ffffff | Sound label text; Play/Stop button text |
| Active state | #4caf50 | Play button background when that sound is currently playing |
| Destructive | not applicable | No destructive actions in Phase 2 |

Accent reserved for: sound label text and Play/Stop button text only.

Active state `#4caf50` (green) is reserved exclusively for: the Play button of a currently-playing sound. It provides unambiguous visual feedback that this specific sound is active. No other elements use this color.

Contrast check: #ffffff on #4caf50 = approximately 3.1:1 (WCAG AA for large text at 24px/600 — passes). #ffffff on #1e1e1e (button default bg) = 14.7:1 (WCAG AAA).

Source: Phase 1 UI-SPEC color contract + active-state color chosen as discretionary default (CONTEXT.md granted Claude discretion over Play/Stop button state management).

## Interaction Contract

### Sound Row Interaction Sequence

1. Page loads → Start overlay visible (Phase 1 component), `#app` hidden
2. User clicks "▶ Start App" → overlay removed, `#app` revealed with `#sound-list` populated
3. All 7 rows render in catalog order; all Play buttons show Idle state; all Stop buttons show inactive state
4. User clicks "▶ Play" on a row:
   - `ensureRunning()` called on AudioEngine
   - Sound's audio node starts (noise generator or sample player)
   - Play button transitions to Playing state (bg `#4caf50`)
   - Stop button transitions to Active state (`opacity: 1`)
5. User clicks "■ Stop" on a row:
   - Sound's audio node stopped/disconnected
   - Play button returns to Idle state
   - Stop button returns to inactive state
6. Multiple sounds can be active simultaneously — each row tracks its own state independently

### State Isolation

Each row manages its own play/stop state independently. Playing Sound A has no effect on Sound B's button states. No "stop all" control exists in this phase.

### No Transition Animations

This is a dev scaffold. Button state changes are synchronous and instant — no CSS transitions or animations. Phase 3 will introduce any needed motion design.

### Focus Management

- No manual focus management beyond native browser behavior
- Interactive elements are native `<button>` elements — keyboard navigation via Tab/Enter/Space works by default

## Accessibility Contract

| Requirement | Implementation |
|-------------|---------------|
| Button roles | Native `<button>` elements (implicit role="button") |
| Play button label | "▶ Play" — visible text; no aria-label needed |
| Stop button label | "■ Stop" — visible text; no aria-label needed |
| Sound label association | `.sound-label` text is adjacent in DOM order to its buttons — screen reader reads row left-to-right naturally |
| Color contrast (label text) | #ffffff on #1e1e1e = 14.7:1 (WCAG AAA) |
| Color contrast (Play idle) | #ffffff on #1e1e1e = 14.7:1 (WCAG AAA) |
| Color contrast (Play active) | #ffffff on #4caf50 = ~3.1:1 (WCAG AA large text — 24px/600 qualifies as large text) |
| Color contrast (Stop inactive) | #ffffff at 0.4 opacity on #1e1e1e — does not meet WCAG; acceptable for dev scaffold only |
| Disabled state | `disabled` attribute on Play button when sound is playing; `disabled` on Stop button when sound is not playing |
| Keyboard activation | Native `<button>` responds to Enter and Space |
| Focus indicator | `focus-visible` outline: 2px solid #ffffff, offset 3px |

Note: The Stop button at 40% opacity in inactive state does not meet WCAG contrast. This is acceptable because this is a throwaway dev scaffold. Phase 3 will re-implement all controls with full accessibility compliance.
